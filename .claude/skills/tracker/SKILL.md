# tracker

`tracker` — это батч-обёртка над скиллом `extract-price`. За один запуск он
проходит по зашитому списку URL, снимает цену с каждой карточки и сводит
результаты в одну таблицу прогона.

Ключевой принцип: **tracker переиспользует `extract-price`, а не дублирует его
логику.** Всё, что касается загрузки страницы, поиска цены, нормализации числа
и валюты и определения рассрочки, живёт в `extract-price`. Здесь — только обход
списка и сборка таблицы.

## Контракт

**Вход:** ничего передавать не нужно — список отслеживаемых URL зашит ниже.

**Выход:** одна таблица прогона. Одна строка на товар (на URL), в строке:

| Поле            | Значение                                                        |
| --------------- | --------------------------------------------------------------- |
| `url`           | URL карточки товара из списка.                                   |
| `regular_price` | обычная цена (число) — из `extract-price`.                       |
| `sale_price`    | цена по скидке (число) или `null`, если скидки нет.              |
| `has_credit`    | `true`/`false` — доступна ли рассрочка.                          |

Это будущая «запись прогона»: на следующих шагах её будем сохранять, сравнивать
с предыдущей (diff) и отправлять в Telegram.

## Отслеживаемый список URL

Список зашит прямо в скилл, чтобы трекер запускался одной фразой
(«запусти tracker»), без ручного перечисления ссылок. Чтобы добавить/убрать
товар — редактируй этот список.

```
https://www.detmir.ru/product/index/id/4576880/
https://www.detmir.ru/product/index/id/4576776/
https://www.detmir.ru/product/index/id/4066114/
https://www.coffee-butik.ru/katalog/kofe/sublimirovannyj-rastvorimyj-kofe/p-gold/
https://www.ozon.ru/
https://www.wildberries.ru/
https://mosprivoz.ru/
https://www.r-ulybka.ru/
```

## Главный сценарий

1. Возьми список URL, зашитый выше.
2. Для **каждого** URL по очереди вызови скилл `extract-price`, передав ему этот
   один URL. Получишь объект `{ regular_price, sale_price, has_credit }`.
   - Не изобретай свою логику парсинга — вся она в `extract-price`.
3. Добавь к результату сам `url` и положи строку в общую таблицу прогона.
4. Если для какого-то URL `extract-price` вернул ошибку (`price_not_found` и
   т. п.) — не прерывай прогон: занеси строку с `regular_price: null` и полем
   `error`, и продолжай со следующим URL. Один сбойный товар не должен ронять
   весь прогон.
5. Когда список пройден — выведи **одну** цельную таблицу по всем товарам.

## Формат вывода

Верни единую таблицу прогона (Markdown-таблица или массив объектов), где ровно
одна строка на каждый URL из списка. Пример:

| url                                             | regular_price | sale_price | has_credit |
| ----------------------------------------------- | ------------- | ---------- | ---------- |
| https://www.detmir.ru/product/index/id/4576880/ | 999           | null       | false      |
| https://www.detmir.ru/product/index/id/4576776/ | 840           | null       | false      |
| https://www.ozon.ru/…                           | 1299          | 649        | true       |

Никакого свободного описания вместо таблицы — результат прогона это именно
таблица «товар → цены».

## Сравнение с предыдущим прогоном (diff + значимость)

После сборки свежей таблицы и **перед** сохранением нового прогона tracker
сравнивает его с предыдущим прогоном из `tracker-data`.

1. **Найди предыдущий прогон.** Через GitHub MCP получи список файлов в корне
   `tracker-data` (`mcp__github__get_file_contents` с `path: "/"`). Файлы
   называются `YYYY-MM-DD.json` — отбери их и возьми **последний по дате**,
   который не является текущим (сегодняшним) прогоном. Прочитай его содержимое
   через MCP.
   - Если файлов-прогонов ещё нет (или есть только сегодняшний) — это **первый
     запуск**, сравнивать не с чем: diff пустой, переходи к сохранению.

2. **Посчитай diff.** Сопоставь товары прошлого и текущего прогона **по `url`**.
   Для каждого совпавшего URL сравни поля `regular_price`, `sale_price`,
   `has_credit` и зафиксируй, что изменилось (старое → новое значение). URL,
   которых нет в одном из прогонов, отметь как `added` / `removed`.

3. **Примени правила значимости из `KNOWLEDGE.md`.** Прогони каждое изменение
   через правила (порог цены ≥ 1%, появление/исчезновение скидки, любое
   изменение рассрочки, новый/пропавший товар) и **отбрось незначимые** (напр.
   колебание цены на копейки). См. соседний файл `KNOWLEDGE.md` — это источник
   правил, держи их там, а не хардкодь в сценарии.

4. **Выведи только значимые изменения.** На выходе — список изменений по
   товарам (URL + что именно изменилось и как), а НЕ весь список цен. Если
   значимых изменений нет — так и скажи: «значимых изменений нет».

Пример вывода значимых изменений:

| url                     | изменение                                   |
| ----------------------- | ------------------------------------------- |
| https://www.ozon.ru/…   | скидка 799 → 649 (−18.8%); рассрочка нет → да |
| https://coffee-butik/…  | цена 1299 → 1149 (−11.5%)                    |

## Сохранение прогона в tracker-data (через GitHub MCP)

Каждый прогон нужно сохранять в отдельный приватный репозиторий **`tracker-data`**
(владелец — тот же аккаунт GitHub), чтобы данные прогонов жили отдельно от кода
скиллов и накапливали историю по датам.

Схема хранения:
- **Один прогон — один файл** `YYYY-MM-DD.json` в корне репозитория, где имя —
  дата прогона (например, `2026-07-06.json`).
- **Содержимое файла** — таблица прогона в формате JSON: массив объектов, по
  одному на товар:

  ```json
  [
    { "url": "https://www.detmir.ru/product/index/id/4576880/",
      "regular_price": 999, "sale_price": null, "has_credit": false },
    { "url": "https://www.ozon.ru/...",
      "regular_price": 1299, "sale_price": 649, "has_credit": true }
  ]
  ```

Порядок действий:
1. Собери таблицу прогона (см. выше) как массив объектов.
2. Определи имя файла по сегодняшней дате: `YYYY-MM-DD.json`.
3. Запиши файл в `tracker-data` **через GitHub MCP**, инструментом
   `mcp__github__create_or_update_file`:
   - `owner` — аккаунт-владелец репозитория, `repo` — `tracker-data`,
     `branch` — `main`, `path` — `YYYY-MM-DD.json`.
   - `content` — JSON прогона (с отступами для читаемости).
   - `message` — например `run: 2026-07-06`.
   - Запись файла через MCP — это **сразу отдельный коммит на GitHub**, без
     локального `git clone` и `git push`.
4. Если файл с сегодняшней датой уже существует (повторный прогон за день) —
   сначала получи его SHA (`mcp__github__get_file_contents`) и передай в
   `sha`, чтобы обновить файл, а не упасть с ошибкой.

Ничего не клонируй локально: и чтение SHA, и запись файла идут напрямую через
GitHub MCP.
